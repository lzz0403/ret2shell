## 五、开发背景与目的

回归终端（英文补充名：Ret2Shell）的开发背景来源于网络安全竞赛和教学训练场景中长期存在的共性问题。传统 CTF 比赛与训练活动通常需要分别处理用户注册、队伍组织、题目发布、靶机环境启动、提交判定、成绩统计、公告通知和赛后归档等多个环节，如果依靠临时脚本或分散工具进行组织，往往会带来配置重复、运维成本高、数据割裂、审计困难和比赛稳定性不足等问题。特别是在多人参赛、题目数量较多、比赛时长较长的条件下，题目环境快速启停、权限隔离、日志追踪和实时榜单展示会直接影响赛事体验和管理效率。

本软件旨在提供一套面向网络安全竞赛与训练活动的统一管理平台，将赛事组织、题目运行和数据管理纳入同一系统中进行集中控制。软件通过标准化的前后端交互界面和可配置的后台服务能力，帮助管理员完成建赛、出题、发布、运行、统计和归档等工作，同时为选手提供统一的注册登录、组队、查看题目、启动实例、提交答案和查看榜单等操作入口。对于高校实验教学、社团训练、企业内部安全演练和公开 CTF 赛事等场景，本软件能够减少人工配置工作，提升平台稳定性，并为后续的活动复盘和资产沉淀提供基础数据支持。

## 六、软件总体设计与系统架构

回归终端采用前后端分离的总体设计思路。前端部分负责页面展示、交互操作、数据可视化和富文本内容渲染，后端部分负责认证鉴权、业务逻辑处理、数据库访问、缓存管理、消息队列通信、题目实例调度、文件存储和审计记录。通过这种分层方式，软件既能够在用户侧提供统一的操作体验，也能够在服务侧保持较高的可扩展性和维护性。

从实现结构上看，软件前端使用 SolidJS、TypeScript 和 Vite 构建，适合实现高交互性的单页 Web 界面。后端采用 Rust 多 crate 工作区架构，并以 Axum 和 Tokio 作为核心网络服务和异步运行基础。数据持久化依赖 PostgreSQL，缓存层使用 Redis 或 Valkey，消息队列使用 NATS。对于需要动态启动题目实例的场景，软件集成了 Kubernetes 集群管理能力，可用于调度题目运行容器、网络暴露和生命周期控制。对于文件类资源，系统通过本地存储路径和专门的 Bucket、Media 模块管理比赛资源、用户上传文件和媒体内容，从而形成较为完整的平台支撑能力。

| 架构层级 | 组成与说明 |
| --- | --- |
| 表现层 | 基于 SolidJS + TypeScript 的 Web 界面，负责用户操作、页面展示、榜单图表、文档渲染和终端交互等功能 |
| 服务层 | 基于 Rust + Axum + Tokio 的后台服务，负责认证、路由、业务处理、接口输出和异步任务调度 |
| 数据层 | PostgreSQL 负责业务数据持久化，Redis/Valkey 负责缓存与会话辅助，NATS 负责异步消息传递 |
| 运行层 | Kubernetes 用于题目实例调度与网络资源控制，支持容器化运行场景 |
| 存储层 | Bucket 模块管理比赛与题目资源，Media 模块管理上传文件与媒体内容 |
| 部署层 | 后端可作为单一二进制程序 `r2s-server` 运行，前端可构建为静态文件并由内置静态服务或 Nginx 提供访问 |

## 七、主要功能说明

### 7.1 用户与身份管理

软件提供面向平台用户的账户体系，支持注册、登录、找回密码、密码重置、邮箱验证、验证码校验和第三方 OAuth 登录等功能。平台能够区分普通选手、队伍成员、比赛管理员和平台运维管理人员等不同身份，并基于角色和权限控制用户的访问范围。通过统一的身份鉴别和权限管理机制，系统可以在同一平台上同时承载选手参赛、管理员建赛和运维管理等多类业务活动。

### 7.2 赛事与训练场管理

软件支持正式比赛与长期训练两类主要场景。管理员可以创建新的赛事或训练空间，配置比赛名称、时间范围、队伍规模、访问策略、规则文档、赛程说明和归档策略等内容。通过统一的赛事实体管理能力，平台可以对不同活动进行隔离管理，并将参赛队伍、题目集合、公告内容和统计数据归属于对应赛事，从而保持数据结构清晰、查询路径明确和赛后归档完整。

### 7.3 题目管理与资源发布

在题目管理方面，软件支持题目创建、分类标签设置、附件发布、提示信息管理、动态分值配置、答案校验和题目描述维护等功能。管理员可以为题目上传说明文档、静态附件或其他必要资源，也可以根据比赛规则设置不同的校验逻辑与展示策略。该模块有助于形成统一的题目录入与发布流程，避免在比赛执行期间出现题目信息分散、资源丢失或版本不一致的问题。

### 7.4 题目实例与集群调度

对于需要运行容器化环境的题目，软件提供题目实例启动、延长、停止和状态管理能力。系统可结合集群模块将题目实例分配到指定节点，并按配置暴露网络访问入口、执行生命周期脚本和维护环境运行状态。该能力适用于 Web、Pwn、Misc 等需要在线靶机或交互式实验环境的赛题场景，能够降低管理员手工创建环境的工作量，并提高比赛期间环境管理的一致性和可追踪性。

### 7.5 提交判定与积分统计

软件支持选手提交答案、记录提交结果、维护解题日志和生成积分榜。系统能够根据题目校验规则判断提交是否正确，并将比赛成绩、队伍排名、解题时间和历史记录进行统一展示。对于采用动态分值的比赛场景，平台还能根据题目被解出情况调整分值并更新榜单，从而满足现代 CTF 比赛对实时性、公平性和观赏性的要求。

### 7.6 公告、知识库与交流辅助

为了满足比赛和训练活动中的内容协同需求，软件提供公告、Wiki、日历和通知等功能模块，用于发布赛事规则、培训材料、时间安排和活动更新。部分场景下还可以借助聊天或交流类功能实现题目讨论和平台内沟通。通过将内容管理能力与核心赛事能力整合在一个系统中，用户无需频繁切换多个外部工具即可完成信息获取和活动参与。

### 7.7 平台运维、审计与扩展管理

软件还提供面向平台维护人员的运维与审计能力，包括日志查看、审计记录、邮件发送、媒体管理、OAuth 配置、集群配置、平台统计和运行状态检查等功能。系统支持统一配置部分运行参数，并可对关键操作进行记录与追踪，以满足平台日常维护、故障定位和赛后复盘的需要。对于需要扩展脚本逻辑的场景，软件还预留了脚本化处理能力，可用于 Checker、流量处理或生命周期脚本等业务扩展。

## 八、主要技术特点

回归终端的一个显著特点是采用 Rust 多模块工作区进行后台实现，将认证、缓存、数据库、邮件、媒体、集群、脚本引擎和消息处理等能力拆分为相对独立的 crate。这样的结构有利于降低模块之间的耦合度，也便于在持续开发过程中进行职责分离、问题定位和独立维护。对于需要长期演进的平台型软件而言，这种模块化组织方式能够提高代码的可读性和工程的可扩展性。

软件在运行模型上采用异步并发架构，后端服务基于 Tokio 异步运行时处理大量网络请求和后台任务，并通过消息队列完成部分异步工作流衔接。这种方式能够较好地适应比赛期间的高并发访问、榜单刷新、日志采集和邮件通知等场景，避免将所有任务阻塞在单一同步处理链路中。与此同时，平台还结合缓存、数据库和消息机制构建较完整的服务支撑体系，使用户访问和后台处理能够保持相对稳定的协同关系。

在题目运行能力方面，软件集成了 Kubernetes 相关管理逻辑，能够围绕容器化题目环境执行节点选择、实例启动、网络暴露和生命周期控制。相较于单纯依赖人工部署或一次性脚本启动的方案，该设计更适合多题目、多队伍和长时运行的网络安全竞赛场景，也更便于后续扩展弹性调度、资源隔离和自动化运维能力。

在扩展能力方面，平台引入了脚本引擎与可编程组件，用于承载 Checker、OAuth 脚本、流量脚本和生命周期脚本等可配置逻辑。通过将部分变化频繁或和业务细节高度相关的逻辑以脚本化方式处理，系统可以在不频繁修改核心服务的情况下实现题目级或场景级扩展。该设计有利于提升平台在不同赛事规则、不同题目类型和不同部署环境下的适应能力。

在平台安全与可观测性方面，软件实现了验证码、人机校验、权限控制、请求追踪、限流、日志记录和审计留痕等功能。系统在接口层支持统一路由、请求标识传播和访问行为记录，在运维层支持日志目录管理和可选日志服务集成，有助于在比赛高峰期进行稳定性保障、问题定位和安全事件排查。前端界面还集成图表、Markdown 渲染、代码编辑和终端交互等能力，使平台兼具管理功能与竞赛使用体验。

## 十、典型操作流程说明

### 10.1 管理员使用流程

管理员登录平台后，可以先创建比赛或训练场并填写基本信息，再根据活动要求导入或创建题目，配置题目分类、附件、提示、分值和运行策略。若题目需要在线实例，管理员还可以为题目设置对应的运行环境和生命周期规则。完成配置后，管理员可发布公告、开放报名或开放访问权限，并在比赛进行期间查看提交记录、榜单变化、系统日志和题目运行状态，必要时执行手动干预和赛后归档。

### 10.2 选手使用流程

选手访问平台后，可完成注册、登录、加入队伍或创建队伍等操作，并在获得比赛访问权限后进入对应赛事页面。选手可浏览题目列表、查看题目说明、下载附件、启动题目实例、获取提示并提交答案。系统会根据提交结果更新解题记录与队伍积分，选手还可以查看公告、规则文档、榜单排名和历史记录，从而在统一界面内完成完整的参赛流程。

### 10.3 运维与维护流程

平台运维人员可结合配置文件、部署脚本和后台管理界面对服务进行部署、升级和巡检。后端程序启动后可加载数据库、缓存、消息队列、媒体存储、脚本引擎和集群等模块，并通过统一路由对外提供接口服务。运维人员可在运行期间查看日志、审计记录、平台统计、外部服务连接状态和集群状态，根据活动规模调整部署参数，并在活动结束后进行数据保留、归档和环境清理。

回归终端是围绕网络安全竞赛与训练业务场景独立设计和实现的软件平台。除依法合规使用的通用开源框架、运行库和第三方依赖外，软件的总体架构设计、业务流程组织、模块划分、接口实现、页面交互、数据模型和核心源代码均由开发者围绕目标场景进行自主设计与开发完成。软件并非对单一现有平台进行简单更名、界面替换或机械复制，而是在赛事管理、题目运行、实例调度、内容协同和平台运维等方面形成了具有完整业务逻辑和工程组织方式的独立软件作品。
